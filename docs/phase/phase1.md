## Phase 1: 초기 세션 기반 인증 시스템 분석

#### 1. 세션 기반 인증

| 항목        | 세부 내용                                                                                | 상태 |
|-----------|--------------------------------------------------------------------------------------|---|
| 세션 인증 구현  | Spring Security 기반 HttpSession 인증 필터 구성,<br/>SecurityFilterChain 설정 - 인증/인가 분리       | ✅ |
| 비밀번호 해시   | 비밀번호 해시  Argon2 해시 적용 단방향 암호화                                                        | ✅ |
| 사용자 정보 관리 | PostgreSQL users 테이블 설계  (id, email, password, role 등)                               | ✅ |
| 인증 흐름 구성  | 로그인 엔드포인트 → 세션 발급, <br/>인증 필터 → SecurityContext 저장 → 권한 검증 흐름, <br/>로그아웃 시 세션 무효화 처리 | ✅ |

#### Phase 0 -> Phase 1 변경 사항

| 구분 | Phase 0 (로컬 메모리) | Phase 1 (Redis 도입) |
| :--- | :--- | :--- |
| **세션 저장소** | 서버 로컬 메모리 | **Redis (외부 저장소)** |
| **지속성** | 서버 재시작 시 모든 세션 유실 | **서버 재시작 후에도 세션 유지** |
| **확장성** | 다중 서버 환경에서 세션 불일치 | **세션 공유로 유연한 확장 가능** |
| **한계** | **서버 장애 시 전체 사용자 로그아웃** | **인프라 분리로 서비스 가용성 확보** |

### [요약]
- **인증 방식**: Spring Security + Servlet Session (JSESSIONID)
- **저장소**: 서버 인메모리 세션
- **비밀번호 암호화**: Argon2 (Argon2PasswordEncoder)
- **응답 형식**: JSON (CustomAuthenticationHandler)

### [결정 배경]
- 초기 프로토타입 단계에서 구현의 단순성과 표준 보안 준수를 우선함
- Argon2를 통해 높은 수준의 비밀번호 보안 확보

### [한계점]
- **확장성**: 단일 서버 구조로, 서버 증설 시 세션 불일치 문제 발생 (Stateless화 필요)
- **성능**: 대규모 트래픽 시 세션 메모리 점유 및 DB 조회 부하 예상

## Phase 1-1: 도메인 모델과 영속성 계층 분리

#### 1. 아키텍처 리팩토링

| 항목 | 세부 내용 | 상태 |
| :--- | :--- | :--- |
| 도메인 모델 분리 | `User` 클래스에서 JPA 어노테이션을 제거하고 순수 도메인 객체(POJO)로 변환 | ✅ |
| 영속성 엔티티 도입 | `UserEntity`를 신설하여 DB 매핑 및 영속성 관리 책임 분리 | ✅ |
| 저장소 계층 추상화 | `UserRepository` 인터페이스를 도메인 계층으로 이동 및 구현체(Impl) 분리 | ✅ |
| 아키텍처 문서화 | 리팩토링된 구조를 반영하여 시스템 아키텍처 다이어그램 업데이트 | ✅ |

#### Phase 1 -> Phase 1-1 변경 사항

| 구분 | Phase 1 (초기 세션 인증) | Phase 1-1 (도메인 분리 리팩토링) |
| :--- | :--- | :--- |
| **객체 모델** | `User` (JPA 엔티티 + 도메인 혼합) | **`User` (Domain) / `UserEntity` (Persistence) 분리** |
| **의존성 방향** | 도메인 로직이 JPA 기술에 강하게 결합 | **인프라(JPA)가 도메인을 의존 (DIP 적용)** |
| **저장소 구조** | `JpaRepository`를 직접 사용 | **도메인 인터페이스와 영속성 구현체로 추상화** |
| **유지보수성** | 기술 스택 변경 시 도메인 코드 수정 발생 | **도메인 로직을 인프라 변화로부터 보호** |

### [요약]
- **구조**: Layered Architecture (Domain - Persistence 분리)
- **객체 변환**: `toDomain()`, `from()` 메서드를 통한 상호 매핑 구현
- **의존성 방향**: Persistence 계층이 Domain 계층을 의존하도록 설계 (도메인 중심)

### [결정 배경]
- 비즈니스 로직과 기술 인프라(JPA) 간의 결합도를 낮추어 유연성 확보
- 도메인 모델이 특정 프레임워크나 DB 기술에 종속되지 않도록 보호

### [한계점]
- 도메인 객체와 엔티티 간의 매핑을 위한 추가적인 코드 및 연산 비용 발생
- 클래스 수 증가로 인한 초기 설계 복잡도 상승
